SECURITY ENHANCED LINUX

1. What is SELinux?

SELinux (Security-Enhanced Linux) is a mandatory access control (MAC) security mechanism implemented in the Linux kernel. It provides a flexible and fine-grained security policy that enforces access restrictions beyond traditional UNIX discretionary access control (DAC).

2. Why Use SELinux?

  • Mandatory Access Control (MAC): Enforces system-wide policies that cannot be bypassed by users.
  • Principle of Least Privilege: Processes run with minimal required permissions.
  • Protection Against Vulnerabilities: Limits damage from compromised services.
  • Fine-Grained Control: Controls access to files, ports, processes, and network interfaces.
  • Type Enforcement (TE): Separates processes into domains and files into types.
  • Multi-Level Security (MLS): Supports classification levels for sensitive data.

3. SELinux Modes

SELinux operates in three distinct modes:

ModeDescriptionBehavior
Enforcing Full security enforcement Policy rules are enforced; violations are denied and logged
Permissive Policy not enforced, only logged Violations are logged but not denied (useful for troubleshooting)
Disabled SELinux is turned off No SELinux checks; no labeling; not recommended
# Check current mode getenforce sestatus # Set mode temporarily (until reboot) sudo setenforce 0 # Permissive mode sudo setenforce 1 # Enforcing mode # Set mode permanently (edit configuration) sudo vi /etc/selinux/config # Change line: SELINUX=enforcing or SELINUX=permissive or SELINUX=disabled # Verify configuration sestatus

4. SELinux Contexts

Every process, file, directory, port, and network interface has an SELinux context (security label). The context format is:

user:role:type:level (optional MLS)

4.1 Context Components

ComponentDescriptionExample
User SELinux user identity system_u, unconfined_u, user_u
Role Defines which domains a user can enter object_r, system_r, unconfined_r
Type Most important for TE (Type Enforcement) httpd_t, httpd_sys_content_t, etc_t
Level MLS/MCS sensitivity level s0, s0-s0:c0.c1023
# View file context ls -Z /etc/passwd # Output: system_u:object_r:passwd_file_t:s0 /etc/passwd # View process context ps -Z ps auxZ | grep httpd # View port context semanage port -l sudo semanage port -l | grep ssh

5. SELinux Policies

SELinux uses policies to define access rules. The main policy types are:

5.1 Targeted Policy (Default)

Targets specific daemons and processes; unconfined processes run without restrictions.

# Check current policy type sestatus | grep "Loaded policy name" # Output: Loaded policy name: targeted

5.2 MLS Policy (Multi-Level Security)

Used for systems requiring strict classification levels (government, military).

5.3 Policy Modules

Policies are built from modules that can be enabled, disabled, or customized.

# List all policy modules semodule -l # Install a policy module sudo semodule -i mymodule.pp # Remove a policy module sudo semodule -r mymodule # Create custom policy module sudo grep httpd /var/log/audit/audit.log | audit2allow -M myhttpd sudo semodule -i myhttpd.pp

6. SELinux Booleans

Booleans are on/off switches that modify policy behavior without writing custom rules.

# List all booleans getsebool -a semanage boolean -l # List booleans related to httpd getsebool -a | grep httpd # Check a specific boolean getsebool httpd_can_network_connect # Set boolean temporarily sudo setsebool httpd_can_network_connect on sudo setsebool httpd_enable_cgi off # Set boolean permanently sudo setsebool -P httpd_can_network_connect on # View boolean with description semanage boolean -l | grep httpd
Common SELinux Booleans:
- httpd_can_network_connect: Allow Apache to make outbound network connections
- httpd_enable_cgi: Allow Apache to execute CGI scripts
- ftp_home_dir: Allow FTP to access user home directories
- samba_enable_home_dirs: Allow Samba to share home directories
- virt_use_nfs: Allow virtualization to use NFS storage

7. SELinux File Context Management

Files and directories must have correct SELinux types for processes to access them.

7.1 View and Change File Context

# View current context ls -Z /var/www/html/ # Change context temporarily sudo chcon -t httpd_sys_content_t /var/www/html/index.html sudo chcon -R -t httpd_sys_content_t /var/www/html/ # Restore default context from policy sudo restorecon -v /var/www/html/index.html sudo restorecon -R -v /var/www/html/ # View default context for a path sudo semanage fcontext -l | grep /var/www

7.2 Add Custom File Context Rules

# Add new context mapping for custom web directory sudo semanage fcontext -a -t httpd_sys_content_t "/custom/web(/.*)?" # Apply the context sudo restorecon -R -v /custom/web # Remove custom context mapping sudo semanage fcontext -d "/custom/web(/.*)?" # View all custom file context rules sudo semanage fcontext -l -C

8. SELinux Port Management

Services are only allowed to listen on specific ports as defined by policy.

# List all ports with their contexts sudo semanage port -l # List ports for specific service sudo semanage port -l | grep http # Add custom port for Apache (e.g., 8080) sudo semanage port -a -t http_port_t -p tcp 8080 # Remove custom port sudo semanage port -d -t http_port_t -p tcp 8080 # Verify port is added sudo semanage port -l | grep 8080
Example: To run Apache on port 8080 instead of 80: sudo semanage port -a -t http_port_t -p tcp 8080

9. SELinux Troubleshooting

9.1 Audit Log Location

# Main audit log (RHEL/CentOS/Fedora) /var/log/audit/audit.log # Alternative location (Debian/Ubuntu) /var/log/audit/audit.log # View SELinux denials sudo ausearch -m avc -ts recent sudo journalctl -t setroubleshoot

9.2 Using sealert (setroubleshoot)

# Install setroubleshoot sudo dnf install setroubleshoot # Fedora/RHEL sudo apt install setroubleshoot # Debian/Ubuntu # View alerts sudo sealert -a /var/log/audit/audit.log # Watch real-time alerts sudo sealert -l /var/log/audit/audit.log

9.3 Common SELinux Denial Analysis Tools

# Search for AVC denials sudo grep "avc: denied" /var/log/audit/audit.log # Generate custom policy from denials sudo grep httpd /var/log/audit/audit.log | audit2allow -M mypolicy sudo semodule -i mypolicy.pp # Show suggested fixes sudo sealert -a /var/log/audit/audit.log | grep "Suggested fixes" # Check if a specific type is defined sudo seinfo -t httpd_sys_content_t -x

11. Common SELinux Scenarios and Fixes

11.1 Web Server Cannot Access Files

# Problem: Apache can't serve files from /var/www/html # Check context ls -Z /var/www/html # Fix: Restore default context sudo restorecon -R -v /var/www/html # Alternative: Change context for custom location sudo semanage fcontext -a -t httpd_sys_content_t "/custom/web(/.*)?" sudo restorecon -R -v /custom/web

11.2 Apache Cannot Connect to Remote Database

# Problem: httpd denied network connection # Solution: Enable boolean sudo setsebool -P httpd_can_network_connect on

11.3 FTP Cannot Access Home Directories

# Solution: Enable boolean sudo setsebool -P ftp_home_dir on

11.4 Custom Port for Service

# Problem: Service can't bind to custom port # Solution: Add port to policy sudo semanage port -a -t http_port_t -p tcp 8080

11.5 NFS Share Cannot Be Accessed by Virtual Machines

# Solution: Enable boolean sudo setsebool -P virt_use_nfs on

12. SELinux Best Practices

  • Never disable SELinux permanently – Use permissive mode for troubleshooting instead.
  • Use booleans first – Before creating custom policies, check if a boolean exists.
  • Restore correct contexts – Use restorecon instead of chcon for permanent fixes.
  • Document custom changes – Keep track of custom contexts, ports, and booleans.
  • Test in permissive mode – Use permissive mode to identify what would be denied.
  • Regular audit review – Monitor /var/log/audit/audit.log for suspicious denials.
  • Use semanage for persistence – Always use semanage for permanent context changes.
  • Keep policy modules minimal – Create specific modules rather than broad rules.
Quick Reference: SELinux Troubleshooting Workflow
1. Check current mode: getenforce
2. View denials: sudo ausearch -m avc -ts recent
3. Analyze with sealert: sudo sealert -a /var/log/audit/audit.log
4. Check if boolean fixes: getsebool -a | grep service
5. Check file contexts: ls -Z /path/to/file
6. Restore context: sudo restorecon -v /path/to/file
7. Create policy if needed: audit2allow -M mymodule
8. Apply module: sudo semodule -i mymodule.pp
